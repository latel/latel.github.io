微证券代码规范

发表于 2016-06-05   更新于 2016-06-30   |   分类于 堆代码的时候   |  

随着项目的推进，和团队成员的增加更替，制定一份规范可以在团队协作中，避免低级 Bug，产出风格统一的代码并有效控制代码质量。
此规范共包含编辑器配置、代码风格规范、注释规范和微证券最佳实践四个方面的内容。

1. 编辑器配置

由于团队成员的个人习惯，每个人都有自己偏爱的编辑器，有的成员甚至会同时使用若干个编辑器，这时候，如果没有仔细配置，每个编辑器的默认行为可能会有差异，譬如，有的编辑器在按TAB时会用TAB缩进，而其他的编辑器可能会用2个或4个空格来代替。幸好，我们有 .editorconfig 可以缓解此现象，目前主流浏览器都支持此全局配置文件，不支持的一般也会有插件来提供支持。

1.1 EditorConfig 是什么？

EditorConfig helps developers define and maintain consistent coding styles between different editors and IDEs.

简言之，EditorConfig 用来定义代码格式，以实现不同编辑器、不同项目成员之间统一代码风格。

EditorConfig 使用 .editorconfig 来存储配置文件。打开一个文件时，EditorConfig 插件会从文件所在目录开始逐级向上查找 .editorconfig，并将规则按照 CSS 类似的规则合并，直到找到包含 root = true 的配置文件为止。

Windows 用户 ：Windows 用户创建配置文件时，需将文件命名为 .editorconfig. ，资源管理器会自动将其重命名为 .editorconfig

2. 编码风格和规范

Javascript 作为一门短时间内创造出来的语言，有着其先天性的缺陷，有些不好的特性不适合在项目中使用。因此有必要做出一些约束来保证这门语言的最佳实践。

2.1 选型

目前业界对此有多个类似但并不完全相同的约束规范。下面将具体分析，选择更加适合我们业务场景的规范。

2.1.1 JSlint

JSLint由Javascript业界大咖Douglas Crockford于2012年开发。

优点：

  • 参数配置完成，可以直接使用

缺点：

  • JSLint不存在配置文件，如果想改变参数设置，那就存在问题
  • 有限的配置选项，许多规则不能禁掉
  • 不能增加个性化规则
  • 没有文档记录规则
  • 很难弄清楚哪个规则引起的错误

2.1.2 JShint

作为一个可配置的JSLint版本，JSHint被开发出来。你可以配置每个规则，将其放到一个配置文件中，这样在大项目中可以容易使用。JSHint对每个规则有好的文档，所以可以准确知道每个规则的作用。将其集成到编辑器也是简单的。

优点：

  • 大多是参数可以配置
  • 支持配置文件，在大项目中容易使用
  • 已经支持许多类库，像jQuery、QUnit、NodeJS、Mocha等
  • 支持基本的ES6

缺点：

  • 难于知道哪个规则产生错误
  • 存在两类选项：强制选项和松散选项。使得配置有些混乱
  • 不支持自定义规则

2.1.3 ESlint

ESLint由 JavaScript 红宝书 作者 Nicholas C. Zakas 编写，是最新出来的工具。它被设计的容易拓展、拥有大量的自定义规则、容易的通过插件来安装。它给出准确的输出，而且包括规则名，这样可以知道哪个规则造成了错误。

优点：

  • 灵活：任何规则都可以开启闭合，以及有些规则有些额外配置
  • 很容易拓展和有需要可用插件
  • 容易理解产出
  • 包含了在其他检查器中不可用的规则，使得ESLint在错误检查上更有用
  • 支持ES6，唯一支持JSX的工具
  • 支持自定义报告

缺点：

  • 需要一些配置
  • 速度慢，但这不是主要问题

支持插件化的 ESLint 灵活性最好，同时可配置项较为丰富，官方文档完善，社区活跃。因此，我们选定 ESLint 作为主要的代码风格检测工具。

2.2 配置

ESLint 由一个主要的配置文件和可选的若干个插件组成，我们应当将此配置文件放置在项目文件夹的根目录。如果某些特别的目录需要额外定制，我们只需在该目录下新增同名配置文件即可。

ESLint 的配置文件主要由三大部分组成，此外还有若干特殊的配置字段。

env

配置代码的目标执行环境。

globals

配置一些额外需要的全局变量名。

rules

具体配置需要的很多规则详情。

请在 http://10.123.10.150/.eslintrc 下载配置文件并放到项目目录的合适位置。

具体的配置还需要在实践中不断总结和优化，本配置将持续保持更新。

2.3 为编辑器添加支持

2.3.1 全局安装 ESLint

首先我们需要安装ESLint的可执行文件，直接使用npm安装即可，如果处于公司内网，也一定不要用 tnpm 来安装，不然 Visual Studio code 的 eslint 插件会提示找不到 ESLint，可以指定代理，或者插个无线网卡连接到 staff-wifi 来临时安装。

使用Unix系全局安装时，可能需要sudo权限

2.3.2 为 SublimeText 添加支持

首先我们需要安装SublimeText的包管理器PackageControl，SublimeText 2/3，都可以去 PackageControl 网站上获取安装指南。安装完成后，我们即可使用 PackageControl 安装所需的如下两个包：

  • SublimeLinter
  • SublimeLinter-contrib-eslint

安装后，即可在 Tools > SublimeLinter 中修改若干配置:

  • 推荐设置 Lint Mode 为 Background，这样就可以在后台静默更新准实时监测了。
  • 推荐设置 Mark Style 为 Fill，获得更明显的提示。

如何使用 PackageControl 安装包，可以参考 https://packagecontrol.io/installation。

2.3.4 为 Visual Studio Code 添加支持

VSC自带插件管理，因此无需像SublimeText一样额外安装插件管理器。在编辑器窗口中按下 CTRL + SHIFT + P(Mac中应为 COMMAND + SHIFT + p)，然后输入ext install并回车，再接着输入插件名，查找到插件后，点插件栏右侧的一朵云状的下载按钮即可下载，需要安装如下两个包：

  • EditorConfig for VS Code
  • ESLint

安装完成后，重启编辑器后即可支持。

我们还提炼了一些注释的约束，希望能在注释层面减少团队的沟通障碍:

  1. 什么时候应该写注释

    如果以后回过头来看这些代码，我能不能一下子明白当初为什么要这样写。

  2. 不要侮辱读者的智慧

    一些很简单很容易看懂的代码是不需要被注释的，注释应当帮助阅读者更好的理解代码中你认为可能会产生费解的部分。

  1. 关注要点

    注释要尽量以最短的原因，提炼要点，而不要写一大堆无关精要的上下文，画蛇添足的注释会造成混乱。

  2. 你写出的每行代码，日后都会由你维护

    当注释代码时，要考虑到不仅将来维护你代码的开发人员要看，而且你自己也可能要看。
    用Phil Haack大师的话来说就是：“一旦一行代码显示屏幕上，你也就成了这段代码的维护者”。
    因此，对于我们写得好(差)的注释而言，我们将是第一个受益者(受害者)。

直接传DOM节点性能更优

避免变量没有定义的报错

缓存节点提高性能,挂载子节点到父节点的变量上更易于理解

合适的留白，增强代码阅读性

考虑以下的代码，合适的留白对于代码的可阅读性是有很大提升的。

尽量不要使用自动格式化插件

自动格式化插件通常都没有被很完善的实现，对于结构已经很好优化过的代码弊大于利，因此在维护的很好的代码文件里不推荐使用。
此类插件更适合用在将混淆的代码重新格式化的场景中。

语义化

考虑到构建工具会自动把非对外的变量名和函数名压缩为单字符，因此我们在编码的时候可以尽量选择长度合适的标识符而无需关心占用问题。我们在编码过程中一定要考虑到标识符的命名是否易于理解和符合情景、符合语义。一个好的标识符可以很轻松的让代码阅读者一眼就知道函数的功能，变量所代表的含义。

定义常量

虽然 ES5 并没有真正意义上的常量，但是我们依然可以使用var来定义全大写字符的变量来代表常量，常量可以很好的提升程序的可阅读性；

链式调用

链式调用每条调用新起一行会是更好的实践，每一行都可以清晰的观察到执行了什么。

对服务器返回的 retcode 处理

不同的后台对于错误的处理不通，错误代码有可能是字符串，也可能是纯数字，所以我们要对这两种情况兼容处理。我们推荐以 recode 和 retmsg 分别指代错误码和错误描述；我们还推荐以 Http 代码为标准来设计 retcode；如果不符合此标准应加注释特别说明，让大家知道不是你写错了。

可插拔

尽量以可插拔的设计来设计一个新的模块，这意味着当该模块不再需要时，也可以很快被移除，而不用对其他模块或调用该模块的业务脚本做出大量修改, 如：
